预设数据管理

确定预设数据的存储位置和内容结构

问题1：预设数据应该存储在哪里？

A

保持在state_machine.c中（推荐）

位置: state_machine.c 中的 SM_ApplyQuickSet() 函数内部（当前实现）

当前代码（state_machine.c:427-436）:
static void SM_ApplyQuickSet(uint8_t preset_index)
{
    const float quick_presets[4][2] = {
        {5.0f, 1.0f},   // 预设1: 5V/1A
        {9.0f, 2.0f},   // 预设2: 9V/2A
        {12.0f, 3.0f},  // 预设3: 12V/3A
        {20.0f, 3.0f}   // 预设4: 20V/3A
    };
    // ...
}

优点

  • 代码简单: 无需新增文件或模块
  • 性能最优: 编译时常量，无运行时开销
  • 符合当前需求: 4个固定预设，不需要动态修改
  • 遵循YAGNI原则: 不过度设计

缺点

  • 修改预设需要重新编译固件
  • 未来如果需要Flash存储，需要重构
B

创建独立模块 quick_set.c

结构: 新增 Layer/Application/quick_set.c 和 quick_set.h

文件结构:
// quick_set.h
typedef struct {
    float voltage;
    float current;
    const char *name;  // 可选：预设名称
} QuickSetPreset_t;

void QuickSet_Init(void);
const QuickSetPreset_t* QuickSet_GetPreset(uint8_t index);
uint8_t QuickSet_GetCount(void);

// quick_set.c
static const QuickSetPreset_t s_presets[4] = {
    {5.0f, 1.0f, "USB"},
    {9.0f, 2.0f, "QC"},
    // ...
};

优点

  • 职责分离，代码组织更清晰
  • 易于扩展（添加更多字段，如预设名称、图标）
  • 未来迁移到Flash存储更容易

缺点

  • 过度设计: 当前需求不需要这么复杂
  • 增加文件数量: 违反简洁性原则
  • 调用链变长: UI → state_machine → quick_set
  • 当前不支持中文，预设名称字段暂时无用
C

从Flash动态加载（未来扩展）

方案: 使用 param_store.c 模块从 W25Q256 Flash读取预设

优点

  • 用户可通过上位机自定义预设
  • 最灵活的方案

缺点

  • 当前阶段不需要: param_store模块尚未完成
  • 复杂度极高: 需要序列化/反序列化、CRC校验
  • 性能开销（Flash读取约2-5ms）
  • 需要设计默认值fallback机制

⚠️ 架构原则提醒

根据项目的 Karpathy编码原则 和 YAGNI原则：

  • 简洁性: 最小化代码行数，避免过度设计
  • 单一职责: 预设数据是状态机的一部分，不需要单独模块
  • 务实优先: 当前需求是固定4个预设，不需要复杂的扩展性

✅ 推荐方案A：保持在state_machine.c中，等到真正需要Flash存储时再重构
