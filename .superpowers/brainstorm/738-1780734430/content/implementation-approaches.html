现在需要选择UI绘制的实现方式

A

独立绘制函数 + 完整刷新（推荐）

实现结构:
// ui_display.c 新增函数

static void UI_DrawQuickSetMenu(uint8_t cursor_pos) {
    // 1. 全屏填充背景色
    // 2. 绘制标题 "QUICK SET"
    // 3. 绘制4个预设项（循环）
    //    - 如果是当前光标位置，绘制粗边框高亮
    //    - 否则绘制细边框
    // 4. 绘制底部提示栏
}

// UI_Display_Process() 中添加状态分支
void UI_Display_Process(void) {
    // ...现有代码...

    // 新增：快速设置子菜单渲染
    if (ui_state == UI_STATE_QUICK_SET) {
        uint8_t cursor = SM_Get_QuickSetCursor();

        // 状态切换时全屏重绘
        if (s_cache.ui_state != UI_STATE_QUICK_SET) {
            UI_DrawQuickSetMenu(cursor);
        }
        // 光标移动时仅重绘变化项
        else if (s_cache.quick_cursor != cursor) {
            UI_RedrawQuickSetItem(s_cache.quick_cursor, 0); // 旧位置
            UI_RedrawQuickSetItem(cursor, 1);               // 新位置
        }

        s_cache.quick_cursor = cursor;
        return;  // 跳过主界面渲染
    }

    // ...原有主界面渲染...
}

优点

  • 代码清晰: 独立函数，职责单一
  • 性能优化: 首次全绘，后续只刷新变化项
  • 易于维护: 子菜单逻辑完全独立
  • 符合Karpathy原则: 简洁、单一职责

缺点

  • 需要额外的缓存变量（s_cache.quick_cursor）
  • 需要两个函数：UI_DrawQuickSetMenu() 和 UI_RedrawQuickSetItem()
工作量估计: ~80行代码
性能: 首次进入86ms（全屏刷新），光标移动10-15ms（仅刷新2项）
B

简化实现 - 每次全部重绘

实现结构:
// ui_display.c 新增函数

static void UI_DrawQuickSetMenu(uint8_t cursor_pos) {
    // 1. 全屏填充背景色
    // 2. 绘制标题
    // 3. 绘制4个预设项（每次都重绘全部）
    // 4. 绘制底部提示栏
}

// UI_Display_Process() 中添加
void UI_Display_Process(void) {
    // ...

    if (ui_state == UI_STATE_QUICK_SET) {
        uint8_t cursor = SM_Get_QuickSetCursor();

        // 光标变化或状态变化时，全屏重绘
        if (s_cache.ui_state != UI_STATE_QUICK_SET ||
            s_cache.quick_cursor != cursor) {
            UI_DrawQuickSetMenu(cursor);
            s_cache.quick_cursor = cursor;
        }
        return;
    }

    // ...
}

优点

  • 最简单: 只需一个绘制函数
  • 无需复杂脏检查: 逻辑直观
  • 代码量最少: ~60行代码

缺点

  • 性能浪费: 每次光标移动都全屏刷新（86ms）
  • 可能闪烁: 取决于刷新速度
  • 不符合项目现有的Dirty-flag优化原则
⚠️ 性能警告: 每次UP/DOWN按键都会触发86ms的全屏刷新，用户体验可能不流畅
C

状态驱动渲染 - 多状态细化

实现思路:
// 将QUICK_SET拆分为多个子状态

typedef enum {
    QUICK_STATE_INIT,      // 首次进入，需要全屏绘制
    QUICK_STATE_BROWSING,  // 正常浏览，增量更新
    QUICK_STATE_APPLYING,  // 应用预设中
} QuickSetState_t;

static QuickSetState_t s_quick_state = QUICK_STATE_INIT;

void UI_Display_Process(void) {
    if (ui_state == UI_STATE_QUICK_SET) {
        switch (s_quick_state) {
            case QUICK_STATE_INIT:
                UI_DrawQuickSetMenuFull();
                s_quick_state = QUICK_STATE_BROWSING;
                break;
            case QUICK_STATE_BROWSING:
                // 仅更新变化部分
                break;
            // ...
        }
    }
}

优点

  • 状态机思维清晰
  • 易于扩展（例如添加"确认"状态）

缺点

  • 过度设计: 当前需求不需要这么多状态
  • 增加复杂度: 引入新的状态枚举和管理逻辑
  • 违反YAGNI原则: 不需要的抽象
  • 代码量最大（~120行）

💡 推荐：方案A - 独立绘制函数 + 完整刷新

这是性能和代码简洁性的最佳平衡点：

  • 符合项目现有的Dirty-flag优化策略
  • 首次进入全绘（86ms可接受），后续光标移动仅10-15ms
  • 代码量适中（~80行），职责清晰
  • 与主界面的渲染逻辑一致
